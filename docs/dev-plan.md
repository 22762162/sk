# 「三鉴」多 AI 协同开发设计方案 V2.1（私人研究版）

**开发态：Claude Code（Fable 5）× Codex 双模型 ｜ 运行态：DeepSeek × GPT × Claude 三模型互证 ｜ 本人签署**

版本 V2.1 · 2026 年 7 月 · 取代 V2.0

---

## 0 版本说明：三个前提

**前提一：项目性质。** 本项目为**个人私有研究项目**——不上架应用商店、不发售、不向任何第三方提供服务，目的是研究"确定性排盘 + 多模型互证"这一技术架构本身。因此 V1.0 中的商业化模块（支付、订阅、双轨合规架构、国内备案模型池、移动端、B 端 API、团队预算）全部移除；合规章节降级为一页私用底线（第 9 节）。

**前提二：开发态与运行态分离（V2.1 核心修正）。** 严格区分两个平面，各自独立选型：

- **开发态（写代码）= 双模型**：**Fable 5**（经 Claude Code，最强大脑，承担高难与高风险工作）+ **Codex**（GPT 系，承担性价比工作、参考实现与异构验证）。Opus 4.8 退出开发角色——Fable 5 能力覆盖其上，同厂商再挂一路属冗余开销，异构信号由 Codex 提供。
- **运行态（应用本体的会诊引擎）= 三模型**：**DeepSeek × GPT × Claude** 作为三位辩手互相验证，回归可行性方案原始设计；私用前提下无备案约束，三家 API 直接调用。

两平面的模型集合有厂商重叠（开发用 Fable 5 / Codex，运行辩手含 Claude / GPT），由此产生的偏袒风险专设防线，见第 2A 节。

**前提三：吸收外部评审。** 本版沿用 V2.0 已吸收的一份外部评审（GPT 出具）的七项主要修正，其中四项核心表述直接替换（见 3.1），评测、隔离、复现、规则库治理均按其建议重构后再按单人规模瘦身。该评审引用的《人工智能拟人化互动服务管理暂行办法》（2026-07-15 施行）已核实为真实法规，但其适用范围为"向境内公众提供持续性情感互动服务"，且明确排除科学研究类服务——本项目不在适用范围内；若未来改变项目性质（分发、收费、开放注册），须先触发第 9.3 节的重新评估闸门。

## 1 开发态角色矩阵：双模型、证据仲裁、人类签署

不使用"三角仲裁"叙事。角色、工具、模型三层解耦，开发态只有两个模型、四个角色、一位签署人：

| 角色 | 工具载体 | 当前模型 | 职责 | 硬约束 |
| --- | --- | --- | --- | --- |
| **主实现者 + 证据分析者** | Claude Code | **Fable 5** | 高难与高风险工作：排盘引擎 Rust 版、规格起草、会诊编排状态机、运行态提示词起草、差分事件证据整理与根因假设、疑难调试 | 不担任自己产出物的评审者；分析涉及自身实现的差分事件时，Codex 必须并行出具独立根因假设 |
| **性价比实现者 + 交叉评审者** | Codex（主仓库工作会话） | GPT 系 | 常规模块实现（Web 前端、后端 CRUD、脚本、测试补写）；全部 PR 风险分诊与严重问题提示；提示词偏袒审计；对抗测试仲裁逻辑 | 触碰过主仓库 L1 代码的会话不得承接参考实现 |
| **参考实现者** | Codex（**独立环境、独立会话**） | GPT 系 | 在独立仓库盲写排盘参考实现（Python）；只见 contract 包 | 环境不挂载主仓库、无主仓库凭据、不复用主仓库工作会话 |
| **最终签署人** | — | 本人 | 所有差分事件裁决、规格批准、规则库合入、发布决定 | 唯一责任主体 |

**性价比分工原则**：Fable 5 的调用集中在错误代价高、推理密度大的任务（第一列职责）；常规实现、样板代码、评审分诊交给 Codex。如需进一步降本，Claude Code 内的琐碎任务可临时切换更低档 Claude 模型执行，不改变角色结构。

三条角色隔离硬规则（写入 `governance/role-separation.yaml` 并由 CI 校验提交元数据）：

```text
author_model != reviewer_model        # Fable 5 写的代码由 Codex 评审，反之亦然
implementer  != sole_analyst          # 差分事件不由单一实现方模型独家分析
final_signer == human                 # 一切裁决由本人签署，AI 只出证据
```

**双模型体系的固有弱点与补偿**：开发态只有两路模型信号，且 Fable 5 兼任主实现与证据分析，对涉及自身代码的分歧存在自利偏差风险。补偿手段有三：L1 差分事件的裁决**首先以外部历源与规范条款为准**，模型分析只是导览；Codex 对每个差分事件并行出具独立假设，两份报告同权重呈交本人；纯推理型、无客观 Oracle 可依的分歧一律升级本人亲自查证，不采信任何一方模型的单独结论。

## 2A 运行态模型矩阵：DeepSeek × GPT × Claude 三师会诊

应用本体的会诊引擎回归可行性方案原始设计，三个异构厂商的模型分绑三个流派：

| 会诊角色 | 模型 | 绑定流派 | 选择理由 |
| --- | --- | --- | --- |
| 辩手 A | Claude（Sonnet 级） | 子平格局派 | 规则遵循与长链条格局判断能力强 |
| 辩手 B | GPT（5.x） | 旺衰扶抑派 | 通用推理与表达均衡 |
| 辩手 C | DeepSeek（V 系） | 调候派（V2 起兼盲派） | 中文古籍语料浸润深、成本极低 |
| 主审仲裁 | 独立实例，**轮换制** | 无流派 | 每次会诊从三家中轮换取一路未参与本场辩论的新实例，不见辩手身份，只依论据质量与规则库校验裁决 |

三轮协议（盲评 → 匿名质证 → 仲裁）、防失效机制（防从众、防幻觉引经、防"自信的说谎者"）、claim 结构化输出均沿用可行性方案第 7 章，叠加本方案 3.3 的 run manifest 与 3.4 的 claim_type。私用前提带来的简化：无需国内备案模型池，三家海外/国产 API 按能力直接选用；推理网关保留多供应商抽象仅为防断供与做模型对照实验。

### 2A.1 两平面交叉污染防线（V2.1 新增，最重要的新治理点）

开发态与运行态的模型集合存在厂商重叠：**为三位辩手写提示词的是 Fable 5（Anthropic），而辩手之一是其同门 Claude**；同理 Codex（GPT 系）参与评审，辩手之二是 GPT。防线四条：

1. **提示词偏袒审计**：Fable 5 起草的全部辩手/仲裁提示词，由 Codex 做盲审计，重点检查三份辩手提示词的信息量、约束严格度、示例质量是否对等（例如是否给 Claude 辩手写了更贴合其习惯的输出格式说明）；审计意见随 PR 归档。
2. **辩手对称性评测**：Change Eval 中固定包含"提示词对称性"检查——将三位辩手的提示词骨架互换模型运行，若某骨架仅在特定模型上显著更优，标记为疑似定制化偏袒。
3. **运行态采纳率监控**：月度评测统计各辩手论断被仲裁采纳率、质证胜率、置信度分布；若 Claude 辩手系统性占优且无法用流派/任务难度解释，触发提示词复查流程。
4. **开发/运行单向阀**：运行态模型永不参与写代码；开发态产出的提示词必须经 Change Eval 闸门才进入运行态，开发会话中的临时 prompt 片段禁止直接上线。

## 2 项目范围重定义（砍什么、留什么、升级什么）

**砍除**：支付与订阅、应用商店上架、国内/海外双轨模型池、Flutter 移动端、营销类功能（分歧转化钩子等）、B 端 API、未成年人模式与公众危机干预流程（无公众用户）、团队与减员测算。

**保留**：五层架构本体（L1 确定性排盘 → L2 规则库 → L3 多模型推理 → L4 仲裁 → L5 呈现）；三条产品铁律（大模型不做历法计算、每句解读可溯源、分歧透明）；会诊直播 UI（作为研究观察窗口保留，可看三模型质证过程）。

**升级为研究核心**：
- **排盘引擎正确性工程**——双实现对拍 + 三层 Oracle，本身就是可写成技术文章的研究产出；
- **多模型辩论协议**——会诊协议的每次迭代配合评测数据，是本项目的主研究对象；
- **巴纳姆自盲实验**——n=1 场景的重新设计见第 10 节，这是全项目最具研究趣味的部分。

**技术栈精简**：L5 改为本地 Web 应用（React/SvelteKit 单页 + FastAPI），部署在本人设备或私有 VPS（加访问认证，不对公网开放注册）。推理网关保留多供应商抽象（防断供、便于换模型做对照实验），但删除地域合规切换逻辑。

## 3 核心工程原则（吸收评审后的定稿表述）

### 3.1 四句定稿表述（替换 V1.0 原文）

一、**关于分歧处理**：任何双实现不一致均生成差分事件（discrepancy event）。AI 可整理证据与提出根因假设，但最终裁决由本人依据规范、原始历源与 ADR 签署；参与实现或规格起草的模型不得担任该事件的唯一分析者。

二、**关于正确性**:双实现与多历源构成**多重佐证信号**，一致仅表示当前观测范围内未发现差异，不单独构成正确性证明。正确性结论必须同时满足：来源独立性审计（各历源无共同数据谱系）、规范条款全覆盖、性质测试通过。

三、**关于可复现**：对托管模型调用追求**可审计、可回放、统计稳定**，不承诺位级可复现。每次调用保存完整 run manifest（见 3.3），温度与种子只是其中两个字段，不是复现保证。

四、**关于评审**：Codex 负责 PR 风险分诊与严重问题提示；普通改动可按风险抽查，L1 计算路径、密钥与数据处理、schema 变更三类高风险 diff 本人必须完整审阅。

### 3.2 三层 Oracle（L1 正确性框架）

| 层级 | 内容 | 交付物 |
| --- | --- | --- |
| **规范 Oracle** | 精确规格：输入输出定义、UTC/UT1/TT 转换规则、ΔT 模型与版本、闰秒数据版本、IANA tzdata 版本、太阳黄经算法版本、支持的日期范围（建议 1600–2200）、模糊/不存在本地时间的处理、舍入规则、流派配置与基础换算的边界；每个条款有编号并映射到测试 ID | `contracts/paipan-spec.md` + 条款-测试映射表 |
| **外部 Oracle** | 经谱系审计的历源（须确认三个历源不共享同一算法/数据源，否则只算一路信号）、天文年历数据、人工核验样本 | `golden-tests/oracle-sources/` + 谱系审计记录 |
| **性质 Oracle** | property-based（proptest）、metamorphic（如"经度东移 15° ≈ 真太阳时 +1h 的关系式"）、边界测试、mutation testing（故意注入错误验证测试网真能捕获） | CI 测试套件 |

### 3.3 Run Manifest（每次模型调用落盘，单人精简版）

```yaml
run_id:                  # 唯一标识
timestamp:
provider: / model_id: / model_release:   # 含服务端版本信息（能取到什么记什么）
input_hash: / system_prompt_hash: / prompt_bundle_hash:
rulebase_commit: / retrieval_manifest:   # 检索到的 chunk id 列表
sampling_parameters:     # 温度、种子等，作为普通字段
output_schema_version:
response_hash: / token_usage: / cost:
```

回归比对不做字符串相等，改测：JSON Schema 合法性、核心 claim 稳定性、规则 ID 引用一致性、无依据论断率、多次调用统计波动带。

### 3.4 Claim 分型（输出协议升级）

在原 claim JSON 基础上增加 `claim_type` 字段，UI 按类型区分渲染：

```json
{
  "claim_type": "computed_fact | school_rule | model_synthesis | reflective_prompt",
  "claim": "...",
  "rule_ids": [], "source_ids": [],
  "confidence": "...", "disagreement": "...", "limitations": "..."
}
```

对应的验收分层原则：**L1 的历法天文换算可以讨论计算正确性；L2–L4 的命理解释只能讨论规则忠实度、一致性与来源，不因多模型一致而宣称预测得到验证。** 这条同时是研究诚实性的底线。

## 5 L1 排盘测试体系（替换"10 万随机时刻"）

测试语料七类，输入空间按 **时间 × 时区 × 经纬度 × DST/历史时区 × 输入精度 × 流派配置 × 距边界距离** 分层：

1. **固定黄金集**（人工可复核，首批 500–1000 例足够单人项目起步，逐步扩充）；
2. **规格条款测试**（每条 spec 条款至少一个测试 ID，覆盖率 CI 统计）；
3. **边界集**（节气交接 ±5 分钟、真太阳时跨日、1988 夏令时、早晚子时、时区重叠与空洞、历史五时区）；
4. **分层随机集**（每晚 CI 按分层配额生成，非均匀随机）；
5. **Metamorphic 测试**（输入变换与输出关系的不变式）；
6. **Mutation 测试**（月度跑一次，验证测试网灵敏度）；
7. **缺陷回归集**（每个差分事件裁决后永久入库）。

对拍协议预定义：哪些字段要求完全相等（四柱、大运序列）、哪些天文量允许 epsilon 及其数值（如节气时刻 ±1 秒）、边界舍入归属、canonicalization 由 contract 包统一提供。

闸门层级：`本地 hook = 快速反馈`；`CI required check = 合入闸门`（黄金集抽样、契约测试、差分测试、schema 校验）；`branch protection = 强制`——单人项目同样开启，required check 不因管理员身份绕过，绕过须在 commit message 记录原因（对自己的纪律约束）。失败用例自动最小化并归档。

## 6 评测体系（单人规模）

| 层级 | 规模 | 触发 | 说明 |
| --- | --- | --- | --- |
| Smoke | 20–30 例 | 每次本地 prompt/编排修改 | 快速回归 |
| Change Eval | 100 例分层案例 | 每次 prompts/ 或会诊协议 PR | 合入闸门 |
| Monthly Eval | 300 例 | 每月 | 趋势监测 |
| Frozen Holdout | 50 例，长期冻结不参与调优 | 重大版本 | 防对测试集过拟合 |

**指标体系**（替换单一"共识率"）——共识率与分歧率保留为**行为描述指标**，不作为优化目标；优化目标为：

```text
schema_valid_rate            # 输出合法率
unsupported_claim_rate ↓     # 无规则依据论断率（核心）
rule_citation_coverage ↑     # 论断引用覆盖率
citation_veracity ↑          # 引用真实性（规则真存在、条件真触发）
appropriate_abstention_rate  # 依据不足时主动弃权率（鼓励弃权而非编造）
cross_run_stability          # 同输入多次调用的统计稳定性
cost_per_report / latency
```

多模型产品的目标不是分歧越少越好，而是：有据处成共识、真实流派差异处保留分歧、缺据处弃权。

**提示词工作流防单模型偏差**（轻量版）：Fable 5 起草 → Codex 做盲测与偏袒审计（按 2A.1 对称性标准，重点检查是否存在对 Claude 辩手措辞习惯的隐性利好）→ Change Eval（含提示词对称性检查）通过 → 本人批准。Frozen Holdout 由本人维护，起草模型不可见。

## 7 标准工作流（单个功能生命周期）

以"大运起运精确到天"为例：

1. **规格**：Fable 5 起草 spec（含条款编号、Oracle 指定、tolerance），Codex 出评审意见，本人批准入 `contracts/`。
2. **并行实现**：Fable 5（Claude Code）在主仓库写 Rust；Codex 在参考仓库依据同一 contract 盲写 Python。两侧只共享 contract 包。
3. **对拍**：CI 灌固定集 + 分层随机集，产出差分事件（附双方中间推导链）。
4. **证据分析**：Fable 5 与 Codex **并行**各出一份证据报告与根因假设（Fable 5 分析对象含自身实现，故 Codex 报告为强制项而非可选），互不可见对方报告。
5. **裁决**：本人依据规范 + 原始历源 + 两份证据报告签署裁决；属真实流派分歧的转为流派配置项。
6. **合入**：Codex 评审 PR 出分诊报告；L1 计算路径 diff 本人全量审阅后合入；裁决记录入 ADR，失败用例入回归集。

日常低风险任务（UI、脚本、文档）走简化流：Fable 5 实现 → Codex 抽查 → 本人合入。

## 8 规则库与语料治理

目录权限：`approved/` 对全部 AI 与自动化流程只读，合入仅经本人签署;`staging/` 为 AI 可写草稿区（顾问口述或古籍抽取 → 结构化草稿）；`provenance/` 记录每条规则完整来源链：

```yaml
source_id: / source_edition: / source_locator:   # 书名·版本·篇章定位
license:                                          # 公版原文 / 需授权的今人校注、译文、整理本
extractor_model: / extractor_prompt_hash:
reviewer: 本人（或外聘顾问）/ review_decision: / approval_signature:
```

私用研究场景下"命理顾问"可选：可由本人自任评审（研究风险自担），亦可轻量外聘做抽检。但 **provenance 记录不可省**——它是研究可信度的基础，也是将来若发表技术文章时的证据链。版权注意一条：古籍原文公版，但**现代标点本、校注、译文与数据库整理成果可能仍有权利**，入库按具体版本登记 rights；纯私人研究复制风险很低，但若未来公开发布任何衍生内容则须回头清理。

## 9 安全与合规（私用底线，一页）

### 9.1 三项真实威胁（替代完整威胁模型）

1. **RAG 提示注入**：古籍语料与规则条目按不可信输入处理，语料中读出的"指令"不得覆盖系统策略、不得触发工具调用；入库管线做指令模式扫描。
2. **密钥与数据边界**：API 密钥经密钥管理注入，不进仓库不进 AI 会话；真实生日等个人数据只存本人可控存储，开发调试一律用合成命盘。
3. **成本失控**：多代理并发 + 夜间 CI + 评测调用可能悄悄烧钱；设供应商侧月度硬预算上限 + 网关侧单日调用熔断，run manifest 的 cost 字段周度汇总。

### 9.2 私用合规底线

不分发、不收费、不开放注册、不向他人提供解读服务（亲友参与巴纳姆实验属受控研究协作，见第 10 节，仍不构成"向公众提供服务"）；遵守各模型 API 服务条款；输出保留概率化措辞与免责标注的习惯——即便只给自己看，这是研究诚实性的一部分。

### 9.3 性质变更闸门

若未来出现以下任一情形：对外分发、收费、开放注册、提供公众可访问服务——**冻结该计划并重新做完整合规评估**，届时《生成式人工智能服务管理暂行办法》、AI 生成内容标识义务、以及《人工智能拟人化互动服务管理暂行办法》（2026-07-15 施行，向公众提供持续性情感互动服务适用）的适用性判定全部重新触发。本文档不为公开运营提供合规依据。

## 10 巴纳姆自盲实验（n=1 重设计）

原方案的双盲对照需要用户群体做统计检验，单人场景不成立，改为两级设计：

**一级 · 自盲混盘测试**（常态化）：系统按随机日程生成报告批次，其中真盘（本人）与干扰盘（随机合成盘 + 历史名人盘）按未知比例混合，隐去排盘细节只呈现解读文本；本人对每条论断盲评"贴合 / 不贴合 / 说不清"，评完揭盲。指标：真盘论断贴合率是否系统性高于干扰盘。单次不显著没关系，跨月累计批次做序贯分析。防作弊要点：干扰盘的日主、格局分布须与真盘同层（否则一眼识破）；揭盲前评分锁定不可改。

**二级 · 小样本亲友盲测**（可选）：邀请 5–10 位知情同意的亲友做同款自盲测试，数据仅本人研究使用、可随时删除。样本仍小，但可把"个性化含金量"从 n=1 轶事升级为方向性证据。

这项实验的研究价值独立于命理效度问题：它测量的是**解读输出的个性化区分度**（反巴纳姆），无论结果如何都是有意义的结论。

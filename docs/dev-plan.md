# 「三鉴」多 AI 协同开发设计方案

**Claude Code（Opus 4.8）× Codex 5.6 三角开发体系**

版本 V1.0 · 2026 年 7 月 · 配套《AI 八字命理 APP 可行性方案》使用

---

## 1 设计总纲：用"多 AI 互证"的方法开发"多 AI 互证"的产品

「三鉴」产品的核心主张是三个异构模型互相质证、分歧透明。本开发方案将同一哲学应用于工程过程本身，形成一个自洽的三角开发体系：

| 角色 | 载体 | 职责定位 | 类比产品架构 |
| --- | --- | --- | --- |
| **主力工程师 + 架构师** | Claude Code（默认模型 + 高难任务切换 Opus 4.8） | 仓库主导权：架构设计、核心实现、重构、测试编写、CI 维护 | 辩手 A |
| **独立交叉验证者** | Codex 5.6（独立环境、独立仓库副本） | 平行实现关键算法、代码评审、差分对拍（differential testing） | 辩手 B |
| **首席仲裁与设计脑** | Opus 4.8（Claude Code 内切换调用，或经 API 单独调用） | 复杂决策：排盘算法边界裁决、会诊协议设计、提示词工程、两方实现分歧时的终审 | 主审仲裁 |

三条开发铁律，与产品的三条产品铁律一一对应：

**铁律一：关键算法必须双实现对拍。** 排盘引擎（L1）这种"错一次就砸口碑"的模块，由 Claude Code 与 Codex 各自独立实现（互不可见对方代码），对同一输入集对拍输出。任何不一致即为缺陷信号，由 Opus 4.8 结合权威历源仲裁。这直接服务于可行性方案中"万例黄金集 100% 通过"的零容忍标准——单一 AI 写代码 + 单一 AI 写测试存在同源盲区，双实现对拍消除同源性。

**铁律二：AI 写代码，人守闸门。** 所有 AI 产出经 PR 流程合入，排盘引擎与合规相关代码必须有人类工程师批准；规则库（L2）内容由命理顾问录入评审，AI 只搭工具链，永不代写命理规则本身。

**铁律三：上下文即资产。** CLAUDE.md、AGENTS.md、子代理定义、Skills 与提示词全部纳入 Git 版本管理，与代码同评审、同回滚。AI 协作配置的质量决定协作产出的质量。

## 2 工具分工与选型理由

### 2.1 Claude Code：仓库的"常驻主力"

Claude Code 是贯穿全程的主开发工具，承担约 70% 的编码量。选择理由与用法要点：

- **终端/IDE 常驻 + 全仓上下文**：适合 monorepo 内跨模块改动（如修改命盘 JSON Schema 后同步更新 Rust 引擎、Python 编排层与 Flutter 模型类）。
- **模型分层使用**：日常实现与测试用默认模型控制成本；涉及节气按秒判界、历史时区规则、会诊状态机设计等高难任务时切换 **Opus 4.8**（`/model` 切换），用最强推理啃最硬的骨头。
- **子代理（subagents）与 Skills**：为项目定制一组专职子代理（见第 4 节），把"排盘专家""合规审查员"等角色固化为可复用配置。
- **Hooks 做纪律执行**：pre-commit hook 强制跑黄金测试集子集；输出过滤 hook 扫描大陆版禁用话术（"预测""改运""消灾"），把可行性方案 10.4 节的伦理红线做进开发流水线。

安装与文档：`npm install -g @anthropic-ai/claude-code`，详见 https://docs.claude.com/en/docs/claude-code/overview （具体配置项以官方文档实时版本为准）。

### 2.2 Codex 5.6：刻意保持"异构"的第二实现者

Codex 的价值恰恰在于它**不是 Claude**——不同训练分布意味着不同的错误分布，这是差分测试有效性的前提。用法要点：

- **独立工作区**：Codex 使用独立的仓库 worktree 或云端沙箱，实现对拍模块时**禁止读取 Claude Code 的实现**（只共享接口规格与测试输入格式），保证实现独立性——这与产品第一轮"盲评"协议同构。
- **PR 评审员**：所有 Claude Code 产出的 PR，由 Codex 做第一轮机器评审（逻辑漏洞、边界遗漏、安全问题），人类工程师只看 Codex 标注后的重点，评审效率提升明显。
- **配置载体**：仓库根部维护 `AGENTS.md`（Codex 的项目说明文件），内容与 `CLAUDE.md` 保持语义同步但独立维护，避免一份配置的措辞偏差同时污染两个 AI。

### 2.3 Opus 4.8：只在刀刃上用的"主审"

Opus 4.8 成本最高、能力最强，限定用于四类场景：架构级设计文档（五层架构的接口契约、会诊状态机）、双实现对拍出现分歧时的算法仲裁、L3/L4 会诊提示词体系的设计与迭代（这本身就是产品核心资产）、以及每阶段末的全仓架构评审。日常琐碎任务不用 Opus，与产品端"必要时才辩论"的降本策略同理。

## 3 仓库结构与 AI 协作脚手架

采用 monorepo，结构围绕五层架构展开：

```
sanjian/
├── CLAUDE.md                    # Claude Code 项目宪法（见 3.1）
├── AGENTS.md                    # Codex 项目说明（语义同步、独立维护）
├── .claude/
│   ├── agents/                  # 项目级子代理定义（见第 4 节）
│   ├── skills/                  # 项目级技能（排盘领域知识、测试规范）
│   └── hooks/                   # 红线话术扫描、黄金集冒烟测试
├── engine-paipan/               # L1 排盘引擎（Rust，Claude Code 主实现）
├── engine-paipan-ref/           # L1 参考实现（Python，Codex 独立实现，仅用于对拍）
├── golden-tests/                # 万例黄金测试集 + 三方历源比对数据
├── rulebase/                    # L2 流派规则库（YAML/JSON 条目 + 古籍分段语料）
├── consult-engine/              # L3+L4 会诊编排（Python/LangGraph + 推理网关）
├── prompts/                     # 辩手/仲裁提示词，版本化管理
├── backend/                     # FastAPI 应用层
├── app/                         # Flutter 客户端
└── infra/                       # CI/CD、可观测（Langfuse 类 trace）、部署
```

### 3.1 CLAUDE.md 核心内容设计（节选骨架）

CLAUDE.md 是 Claude Code 每次会话自动加载的项目记忆，必须把可行性方案的工程铁律翻译成 AI 可执行的指令：

```markdown
# 三鉴项目宪法

## 不可违反的规则
1. engine-paipan 是零 AI 依赖的确定性计算库：禁止在其中引入任何
   LLM 调用、网络请求或非确定性逻辑（含系统时钟依赖）。
2. 修改 engine-paipan 任何计算路径后，必须运行
   `make golden-smoke`（黄金集抽样 500 例），失败禁止提交。
3. 禁止修改 rulebase/ 下的规则条目内容——规则由命理顾问经
   评审流程录入，你只能修改其 schema、校验器与工具链。
4. 面向用户的任何文案：大陆版渠道禁用「预测/改运/消灾/化解/
   必然/注定」，全部论断使用概率化措辞（倾向/提示/或有）。
5. 命盘 JSON Schema 变更属于跨层破坏性变更，必须先出 RFC，
   经 Opus 架构评审后才可实施。

## 工程约定
- Rust：clippy pedantic 级零警告；核心历算函数必须有
  property-based 测试（proptest）。
- 提示词文件（prompts/）视同代码：改动走 PR，附回归评测结果。
- 每次会诊调用四元组落盘：温度0+种子+规则库版本+提示词版本。
```

`AGENTS.md` 包含同样的五条铁律，另加一条 Codex 专属纪律："实现 `engine-paipan-ref` 时禁止查看 `engine-paipan/` 目录，接口契约以 `docs/paipan-spec.md` 为唯一依据。"

## 4 项目子代理设计（.claude/agents/）

为 Claude Code 定制五个专职子代理，把重复性协作模式固化：

| 子代理 | 触发场景 | 职责与工具限制 |
| --- | --- | --- |
| `paipan-dev` | 排盘引擎开发 | 只允许访问 engine-paipan 与 golden-tests；内置寿星算法、VSOP87、IANA tzdata 领域知识摘要；每次改动自动附边界用例 |
| `test-guardian` | 任何 PR 前 | 生成/补充测试，重点覆盖附录 B 的八类边界（节气交接±5分钟、真太阳时跨日、1988 夏令时、早晚子时双盘等）；对测试覆盖率下降的改动直接驳回 |
| `debate-orchestrator` | L3/L4 开发 | 维护三轮会诊状态机与结构化输出协议（claim JSON Schema）；负责温度 0 + 固定种子的可复现性检查 |
| `prompt-engineer` | prompts/ 改动 | 修改辩手/仲裁提示词后自动跑 20 例回归盘，比对共识覆盖率与一致率变化，出差异报告 |
| `compliance-auditor` | 发布前 & 文案改动 | 扫描全部用户可见字符串：大陆版红线词表、免责声明存在性、未成年人拦截逻辑、危机识别转介路径完整性 |

Codex 侧对应配置两个固定任务模板：`ref-implementer`（盲实现参考版）与 `pr-reviewer`（按检查清单评审 Claude 的 PR：边界条件、错误处理、时区陷阱、注入风险）。

## 5 五层架构的任务分配矩阵

| 层 | 模块 | Claude Code（主力） | Codex 5.6 | Opus 4.8 |
| --- | --- | --- | --- | --- |
| L1 | 排盘引擎 | Rust 主实现：节气历算、真太阳时、四柱推导、大运起运 | **Python 盲实现参考版**，供万例对拍 | 对拍分歧仲裁；算法规格评审 |
| L1 | 黄金测试集 | 测试框架 + property-based 测试 | 独立生成边界用例集（与 Claude 生成的用例集互补去同源） | 设计用例覆盖策略 |
| L2 | 规则库工具链 | Schema、校验器、Git 式评审流程、向量入库管线 | 工具链代码评审 | 规则条目结构设计（条件谓词语言的表达力评估） |
| L3 | 推理网关 | 多供应商抽象、国内外模型池热切换、缓存策略 | 评审 + 故障注入测试（模拟断供/超时/降级） | — |
| L4 | 会诊引擎 | LangGraph 状态机、匿名质证、仲裁流程实现 | 评审 + 对抗测试（构造"自信的说谎者"用例攻击仲裁逻辑） | **提示词体系设计**（辩手人设、质证协议、仲裁标准）——这是产品核心资产，用最强模型 |
| L5 | Flutter 客户端 | 排盘可视化、会诊直播 UI、订阅支付 | 分担独立页面模块（学习模块、档案页），按接口契约并行开发 | — |
| 横切 | CI/CD 与可观测 | GitHub Actions、Langfuse 接入、黄金集 CI 阻断 | — | — |
| 横切 | 合规审查 | compliance-auditor 子代理日常执行 | 独立复扫（双引擎降漏检率） | 双轨版本话术策略评审 |

关键点说明——**排盘引擎的双实现对拍**是本方案对可行性方案最大的工程增强：可行性方案要求与"三方权威历源"交叉核对，但权威历源只能覆盖常规日期的抽样比对；双 AI 盲实现可对**任意随机时刻**做全量对拍（每晚 CI 跑 10 万随机时刻），把边界缺陷的发现从"抽检"升级为"地毯式"。两实现 + 三历源共五个独立信号源，一致性即正确性的强证据。

## 6 标准开发工作流（单个功能的生命周期）

以"大运起运精确到天"功能为例，走一遍完整流程：

**第 1 步 · 规格（人 + Opus 4.8）**：产品/工程在 Claude Code 中切换 Opus，用 Plan Mode 产出 `docs/specs/dayun-spec.md`——数学定义（三天折一岁的精确折算规则）、顺逆排判定、边界枚举（出生即交节、跨年交节）、接口契约。人类审定后合入。

**第 2 步 · 并行实现（Claude Code ∥ Codex）**：Claude Code（paipan-dev 子代理）在 `engine-paipan` 实现 Rust 版；同时 Codex 依据同一份 spec 在 `engine-paipan-ref` 盲写 Python 版。两边使用 git worktree 隔离，互不可见。

**第 3 步 · 对拍（CI 自动）**：CI 对两实现灌入黄金集 + 10 万随机时刻。不一致案例自动归档为 `discrepancy-issue`，附双方中间计算过程。

**第 4 步 · 仲裁（Opus 4.8）**：对每个分歧案例，Opus 收到双方实现的推导链 + 权威历源数据，裁定哪方错误并定位根因；无法裁定的（真实流派分歧，如早晚子时）转为流派配置项——与产品 5.2 节"争议规则工程化"完全同构。

**第 5 步 · 评审合入（Codex → 人）**：修正后的 PR 由 Codex 做首轮评审出标注报告，人类工程师终审合入。compliance-auditor 与 test-guardian hook 在合入前自动放行检查。

**第 6 步 · 留痕**：spec 版本、双实现 commit、对拍报告、仲裁记录四件套归档——既是工程资产，也是将来"公开比对工具 + 缺陷赏金"（风险矩阵中排盘舆情的缓解措施）的公信力素材。

日常小任务（UI 调整、文档、脚本）不走全流程，Claude Code 直接实现 + Codex 评审 + 人合入即可。全流程仅用于 L1 计算路径与 L4 仲裁逻辑这两类"错误代价极高"的模块。

## 7 提示词与规则库的 AI 协作纪律（易翻车区专门约定）

这个项目有个特殊性：**提示词和规则库本身就是产品**，而 AI 工具天然倾向于"顺手优化"它们，必须显式设防。

**提示词（prompts/）**：只允许 Opus 4.8 会话产出改动草案，且每次改动必须附 prompt-engineer 子代理的 20 例回归报告（共识覆盖率、同盘一致率、分歧率三指标与基线对比）。指标劣化的改动 CI 拒绝。Codex 不参与提示词编写（避免 GPT 系模型的措辞习惯渗入给 GPT 辩手的提示词，造成隐性偏袒——这是多模型产品特有的污染风险）。

**规则库（rulebase/）**：三个 AI 一律只读。AI 可以做的：schema 校验器、录入辅助工具（顾问口述→结构化草稿，但草稿必须标记 `draft_by_ai: true` 且经顾问逐条确认）、古籍 OCR 与分段入库管线。AI 不可以做的:直接新增/修改规则条目、调整强度权重（权重更新走 8.3 节的贝叶斯建议 + 人工评审通道）。

**古籍语料**：《子平真诠》等典籍为公版古籍可全文入库；当代著作与盲派整理材料涉及版权，入库前须取得授权，AI 管线中加来源白名单校验。

## 8 阶段计划映射（对齐可行性方案 P1–P3）

| 阶段 | AI 协同重点 | 团队形态变化 |
| --- | --- | --- |
| **P1 · MVP（1–3 月）** | 第 1 周搭好 CLAUDE.md/AGENTS.md/子代理脚手架（先建协作基建再写业务代码）；排盘引擎双实现对拍是本阶段核心战役；单模型速览解读用简单管线快速上线 | 原方案后端 2 人中"专职排盘引擎"1 人转型为"排盘引擎 AI 结对主管"：人定规格与仲裁终审，AI 产码。核心团队可从 6–7 人压至 **4–5 人**（省 1 后端 1 前端） |
| **P2 · 会诊版（4–8 月）** | Opus 主导会诊提示词体系设计与迭代；Codex 对抗测试仲裁逻辑；Flutter 双 AI 按页面模块并行；规则库工具链交付给顾问使用 | 算法/提示词工程师 + Opus 结对成为关键产能；顾问开始高频使用 AI 录入辅助工具 |
| **P3 · 数据版（9–14 月）** | 反馈闭环数仓与看板由 Claude Code 快速搭建；双盲对照实验的分组、发放、统计脚本 AI 产出但实验设计人工终审；B 端 API 文档与 SDK 由 AI 生成维护 | 工程重心从"建设"转向"评审 + 实验"，AI 承担绝大部分维护性编码 |

**工具成本粗估**（列入原方案"API 与云基础设施 30–60 万元"预算内，不新增大项）：Claude Code 团队订阅（含 Opus 用量）+ Codex 订阅 + 评测用 API 调用，按 4–5 名工程师配置全年约 **8–15 万元人民币**量级，具体以各厂商当期定价为准。相对其替代的 1–2 名全职人力成本（一线 40–80 万/人/年），投入产出比清晰。

## 9 风险与纪律清单（AI 协同开发特有）

| 风险 | 场景 | 缓解 |
| --- | --- | --- |
| 双实现同源失效 | Codex 偷看 Claude 实现，或两边共用同一份有错的 spec 示例 | 工作区隔离硬约束；spec 只给定义不给实现示例；定期人工抽查参考实现的独立性 |
| AI 幻觉出"看似合理的历法规则" | 如编造某年夏令时规则 | 历法事实一律以 IANA tzdata 与权威历源数据文件为准，禁止 AI 从记忆写历法常量；CLAUDE.md 明文规定 |
| 提示词隐性漂移 | 多次"小优化"累计改变会诊行为 | 提示词版本化 + 每次改动强制回归评测 + 每月对比首版基线 |
| 上下文污染 | 把用户真实命盘数据粘进 AI 开发会话 | 开发一律用合成命盘；生产数据访问与 AI 开发环境物理隔离 |
| 过度依赖单一 AI 判断 | Opus 仲裁也可能错 | 对拍分歧的最终裁决要求"Opus 结论 + 权威历源证据"双满足；纯推理无法验证的分歧升级人类 |
| 合规扫描漏检 | 红线词变体、谐音 | Claude 与 Codex 双引擎独立扫描取并集；上线前人工全文抽检；词表随监管通报月度更新 |
